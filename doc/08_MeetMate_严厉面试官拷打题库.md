# MeetMate 严厉面试官拷打题库

> 面试风格：质疑简历真实性、追问保证边界、要求比较替代方案，并持续追到代码、数据库和故障场景。
>
> 代码基线：2026-07-17 当前工作树，Spring Boot 4.1 + Spring AI 2.0 Java 单体。
>
> 使用方法：先只看“面试官”，脱稿回答；再对照“参考答案”。回答中出现“保证、高并发、最终一致、不会丢”时，必须主动说明作用范围和验证证据。

## 1. 评分规则

严厉面试官不会因为你背出了 Redis、RabbitMQ、Spring AI 就给分。每题按下面标准判断：

| 表现 | 评价 |
|---|---|
| 只复述技术名词 | 不合格，说明没有理解问题 |
| 能解释正常流程 | 勉强合格，但缺少工程判断 |
| 能说清保证范围和失败窗口 | 合格 |
| 能比较替代方案并解释取舍 | 良好 |
| 能给出数据库约束、状态迁移、指标和故障测试 | 优秀 |

统一回答结构：

> 业务问题是什么 -> 当前实现怎么做 -> 它准确保证什么 -> 还存在哪个失败窗口 -> 生产化怎么补 -> 用什么指标验证。

## 2. 第一轮：先判断你的简历是不是吹出来的

### Q1：别念技术栈。十秒钟说清楚这个项目到底解决什么问题。

**参考答案：**

> MeetMate 是一个“从种草到成行”的本地生活协作平台：用户通过探店内容发现商家并预订限量权益，多人场景下可以提交各自偏好，由受 Java 规则约束的 Agent 生成可确认的聚餐方案。

**挂点：** 一上来回答“用了 Spring Boot、Redis、RabbitMQ”，说明你说不清产品问题。

### Q2：这不就是黑马点评换皮吗？你到底写了什么？

**参考答案：**

> 基础商铺、探店和秒杀链路来自经典学习项目，我不回避来源。我的主要增量是多人房间、偏好草稿与确认、PlanRun/Attempt/Event 状态模型、数据库 Dispatcher、Spring AI 只读工具、Java Policy 校验和 SSE 事件恢复；同时我对原秒杀、缓存、登录和 Feed 做了边界审计，能指出已实现能力、现存缺口和可验证的改造方案。

**继续追杀：** “原项目代码和你的代码各占多少？”

> 我不会用行数包装贡献，而会按业务边界说明：商铺和基础社交链路是基线，`meet_*` 数据模型、规划执行链和 AI Harness 是新增；对于旧模块，我能指出现有缺陷、修改方案和验证方法。

### Q3：你写“社交推荐”，推荐算法在哪里？

**参考答案：**

> 当前实现准确地说是“关注关系驱动的社交内容分发”，不是个性化推荐。Feed 通过关注关系写扩散到 Redis ZSet，热榜只是按点赞数排序。若简历写“社交推荐”会过度承诺，我会改成“社交 Feed/内容分发”；真正推荐还需要召回、特征、排序、曝光反馈和离线评估。

### Q4：你说短信登录，短信供应商代码在哪里？

**参考答案：**

> 当前学习版没有接真实短信供应商，只生成验证码写 Redis，并在开发日志中输出。因此简历应写“手机号验证码登录”，不能写“接入短信平台”。生产化需要 Provider 抽象、开发 Fake、发送限频、错误次数和一次性消费，日志不能记录验证码。

### Q5：你写 Caffeine L1 + Redis L2，可我没看到业务使用 Caffeine。你是在撒谎吗？

**参考答案：**

> 当前仓库只创建了 Caffeine CacheManager，业务没有真实 get/put/evict，也没有本地缓存失效广播，所以不能把“多级缓存”写成已完成。我当前可以讲 Redis CacheClient、空值缓存和逻辑过期；完成 L1 读写、多实例失效和测试后，才升级简历表述。

### Q6：460ms 到 135ms 怎么测的？机器、并发、P95 呢？

**参考答案：**

> 如果不能提供测试环境、并发数、总请求量、库存量和统计口径，我不会把这个数字写进最终简历。异步化确实把数据库事务移出请求链路，但需要用固定环境重新记录平均、P95/P99、TPS、错误率、超卖数、重复单和 MQ 收敛时间。

### Q7：这是个人项目，为什么文案像支撑过亿流量？

**参考答案：**

> 我不会宣称真实生产流量。项目价值是用可控实验研究高并发入口、缓存并发和异步恢复：给定请求量和故障点验证不变量，而不是声称有商业用户规模。简历中的数据必须来自可复现压测。

### Q8：你最希望我追问哪个点？别说都可以。

**参考答案：**

> 如果是 Java 后端岗位，我希望讨论秒杀中 Redis、MQ、MySQL 的事实边界；如果是 AI 应用岗位，我希望讨论为什么模型只能做语义、Java 如何通过状态机、只读工具和 Policy 控制最终结果。这两个点最能体现我的技术判断。

## 3. 第二轮：你这到底算不算 Agent

### Q9：你用了 Spring AI 调一次模型，就敢叫 Agent？

**参考答案：**

> 单次 Chat Completion 不能自动称为 Agent。这里之所以称为受控 Agent，是因为 Java 先创建有限执行计划，模型在方案生成阶段使用请求级工具，根据候选详情和草稿校验 Observation 选择餐厅、修订并输出结构化方案；同时具有工具权限、调用预算、终止条件和持久运行状态。它不是开放自治 Agent，而是有界 Agent。

### Q10：Plan-and-Execute 体现在哪里？别给我背定义。

**参考答案：**

> `MeetPlanningAgent` 明确定义四个阶段：读取已确认偏好、召回可行餐厅、查看候选详情、生成并校验方案。Java 创建计划并负责 preflight 和最终 Policy，模型只在受控阶段执行工具调用，因此是 Java 外层 Plan-and-Execute。

### Q11：ReAct 体现在哪里？工具顺序都被你限制了，还有什么自主性？

**参考答案：**

> 前提基本成立：Java 用前驱检查强制“读偏好 -> 找候选 -> 查详情 -> 校验草稿”的阶段顺序。局部自主性体现在模型可以决定检查候选集中的哪些餐厅、是否分批补查，以及根据校验 Observation 如何修订草稿；它不能发明新工具、跳过必要阶段或无限循环。`maxToolCalls`、必要工具集合、已检查候选和已验证草稿指纹都由 Java 记录。这里刻意牺牲自治程度换取可验证性，所以我称它“受控 ReAct”，而不是全自治 ReAct。

### Q12：既然步骤固定，为什么不用纯 Java 工作流？

**参考答案：**

> 候选召回、预算距离过滤和最终校验确实应该用 Java。模型的价值在于把多人自然语言偏好结构化、解释冲突、总结软偏好取舍和组织用户可读方案。纯 Java 规则适合确定性计算，不适合覆盖自然语言表达的组合；模型不应该接管数值排序和权限。

### Q13：模型究竟能决定什么，不能决定什么？

**参考答案：**

> 模型能决定方案文案、软偏好取舍表达、建议时间组织和在候选集内选择组合；不能决定用户身份、房间成员、硬约束是否生效、候选事实、方案是否合法或最终确认。后者全部由 Java 和 MySQL 状态机掌握。

### Q14：为什么要求恰好一个首选和两个备选？这是技术约束还是拍脑袋？

**参考答案：**

> 这是产品和可验证性约束。固定三个方案能限制 Token 与落库数量，也能让 `uk_proposal_run_rank`、前端展示和 Policy 校验简单明确。它不是通用最优值，若产品需要动态数量，应把数量作为受限配置并同步修改输出 Schema、校验和索引语义。

### Q15：你的 Agent 有记忆吗？

**参考答案：**

> 没有模型侧长期记忆。每个 Attempt 创建新的工具实例，运行事实存 MySQL；澄清后创建新 Attempt，重新加载已确认偏好和授权后的约束。这样恢复依赖业务事实而不是隐藏模型上下文，更容易审计和重放。

### Q16：那这和普通异步工作流有什么区别？

**参考答案：**

> 外层确实是确定性工作流，这是有意设计。区别在于方案阶段模型会根据 Observation 做有限工具决策并生成结构化候选方案。我的重点不是证明“越自治越 Agent”，而是证明不确定组件可以被确定性 Harness 管住。

### Q17：为什么不用一个大 Prompt，把所有数据一次性塞给模型？

**参考答案：**

> 大 Prompt 会扩大上下文、暴露无关数据并弱化权限边界，也无法证明模型确实查看了所选候选。请求级工具按需返回当前房间数据，Java 还能记录已调用工具、已检查候选和调用预算。代价是多轮调用延迟更高，因此工具数量必须有限。

### Q18：为什么最终还要再做一次 Java Policy？模型不是已经调用 validate 工具了吗？

**参考答案：**

> 工具校验发生在模型循环内，模型可能校验 A 草稿后返回 B 草稿。项目记录成功校验草稿的指纹，并要求最终返回与已校验草稿一致；随后 `MeetPlanPolicyService` 再检查候选归属、数量、唯一性和字段。内部工具校验提升反馈，外部 Policy 才是最终信任边界。

## 4. 第三轮：为什么选 Spring AI

### Q19：为什么选 Spring AI，不选 LangGraph？

**参考答案：**

> 当前流程只有一个受控工具循环和最多一次澄清，澄清后重新创建 Attempt，不需要恢复图节点内部上下文。Spring AI 与现有 Java 单体共享 DI、配置、DTO、事务和业务规则，交付成本更低。若未来出现多轮投票、多个中断点、跨天恢复和复杂分支，再评估 LangGraph 的图与 Checkpoint 能力。

### Q20：你以前简历写 LangGraph，现在又改 Spring AI，是不是技术摇摆？

**参考答案：**

> 是需求收敛后的架构调整，不是为了追热点。早期设想包含多轮投票和长流程中断，LangGraph 合适；MVP 收敛为一次澄清和 Java 事实状态后，独立 Python 服务、跨服务鉴权与 Checkpoint 的成本超过收益，所以改成 Spring AI 单体，并通过 ADR 记录了取舍。

### Q21：为什么不用 LangChain4j？

**参考答案：**

> LangChain4j 也是可行的 Java 方案。选择 Spring AI 的原因是项目已经使用 Spring Boot，希望直接复用配置绑定、Bean 生命周期、ChatClient 和 Tool Calling，并减少另一套集成风格。这个选择不是性能或能力上的绝对优越，而是生态一致性和交付成本取舍。

### Q22：为什么不用自己调 OpenAI HTTP API？

**参考答案：**

> 自己调用能减少框架依赖，但需要重复处理模型配置、结构化映射、工具描述、调用循环、重试和多供应商差异。Spring AI 负责接入层，业务 Harness 仍由我自己控制。若框架抽象阻碍关键能力，可以在 ChatModel 边界替换实现，而不是把业务逻辑耦合到 HTTP JSON。

### Q23：为什么不用 MCP 暴露工具？

**参考答案：**

> 当前工具只服务同一 Java 进程、强绑定一次 room/run/attempt，并直接复用业务服务，进程内工具更简单。MCP 更适合多个 Agent 或外部客户端共享远程工具、需要标准化发现和跨进程权限时。现在引入会增加网络、认证和部署面，没有足够收益。

### Q24：你怎么避免被 Spring AI 框架锁死？

**参考答案：**

> 业务状态机、候选服务、Policy、DTO 和持久化不依赖具体模型；Spring AI 只位于 `MeetPlanningAiService` 和工具注解边界。更换模型供应商通过 OpenAI-compatible 配置完成，更换框架则重写模型适配层，不需要推翻 Run/Attempt 和业务规则。

### Q25：为什么使用 OpenAI-compatible 接口？

**参考答案：**

> 它让模型、base URL 和密钥通过环境配置注入，可以连接多个兼容供应商，减少业务代码分支。但“兼容”不代表所有 Tool Calling、JSON Schema 和错误语义完全一致，因此仍需要供应商契约测试和降级策略。

### Q26：Spring AI 的价值是什么？你的价值又是什么？

**参考答案：**

> Spring AI 解决模型接入、结构化映射和 Tool Calling。我的工程工作是业务上下文裁剪、工具权限、状态持久化、CAS、调用预算、结果 Policy、重试分类、SSE 事件和测试。只会调用框架 API 不是这个项目的核心。

## 5. 第四轮：Human-in-the-loop、幻觉与安全

### Q27：Human-in-the-loop 到底怎么实现？不会是前端弹个框就算吧？

**参考答案：**

> 候选不足时，执行服务在短事务内将 Attempt、Run 和 Room 原子迁移到 `WAITING_INPUT`，写入 Clarification 和持久事件，然后结束模型调用。只有目标成员能回答；同意放宽后更新业务事实、在同一 Run 下创建新 Attempt 并重新投递，不保持 HTTP 连接或 JVM 线程等待。

### Q28：为什么不恢复上一次模型上下文？

**参考答案：**

> 业务事实可能已经变化，而且隐藏上下文难审计。新 Attempt 从 MySQL 重新读取已确认偏好和约束，输入可重建、结果可比较。当前只有一次澄清，重新调用的成本可接受；复杂多轮场景才需要 Checkpoint。

### Q29：为什么只允许一次澄清？

**参考答案：**

> 为了终止性、成本和安全。无限澄清可能逐步侵蚀成员硬约束，也难预测调用成本。一次仍无可行候选就明确失败；过敏原等风险约束永不自动放宽。

### Q30：模型如何知道该问谁？

**参考答案：**

> 不是模型决定。Java 根据过滤原因和已确认偏好选择可安全放宽的目标，例如预算冲突选择最严格预算成员；模型只润色问题，不能改变 targetUserId、constraintKey 和 options。

### Q31：怎么防止模型编造餐厅？

**参考答案：**

> Java 先召回并过滤候选，工具只返回候选集；最终 shopId 必须属于候选、所选餐厅必须被详情工具检查、草稿必须通过工具校验且指纹一致，落库前和房主确认时再次校验。模型没有任意查询或写库能力。

### Q32：Prompt Injection 怎么防？不要只说 system prompt。

**参考答案：**

> Prompt 只是软防线。硬防线是把商铺和偏好当不可信数据、工具请求绑定当前房间、只开放读和校验、限制调用次数与输入输出长度，并由 Java 校验候选归属和状态迁移。即使模型被诱导，也没有写权限。

### Q33：模型返回合法 JSON 就可信了吗？

**参考答案：**

> 不可信。结构化输出只解决语法，不能保证事实正确或业务合法。DTO 反序列化后仍要做字段、数量、候选归属、唯一性、硬约束和状态检查。

### Q34：模型挂了，整个产品是不是不能用？

**参考答案：**

> 不会让整个系统不可用，登录、商铺、Feed 和秒杀不依赖模型。偏好解析在 AI 禁用或调用失败时，会保留原文并生成可编辑草稿；方案生成只有 AI 被显式禁用时才按 Java 候选排序生成确定性三方案。AI 已启用但调用失败时，当前实现只对瞬时异常受限重试，最终会把 Attempt、Run 和 Room 标记为 FAILED；仓库还没有将 FAILED 房间重置为 READY_TO_PLAN 的重规划入口，这是恢复缺口。生产化应增加熔断后的确定性降级或受控重规划，但不能把它说成现有能力。

### Q35：Agent 怎么评测？别给我说“感觉效果不错”。

**参考答案：**

> 当前自动化测试已经覆盖候选归属、重复 shopId、字段边界、必要工具、调用预算、草稿指纹，以及 Dispatcher/执行状态 CAS；这只能证明 Harness，不代表模型效果好。模型层还需要建立固定冲突数据集，记录结构化成功率、候选幻觉数、规则拒绝率、平均工具次数、人工采纳率和成本。真实模型评测放离线任务，CI 使用 Fake/Stub Model 做确定性契约测试。

## 6. 第五轮：RabbitMQ 选型与可靠性

### Q36：为什么选 RabbitMQ，不选 Kafka？

**参考答案：**

> 当前主要是订单命令和后台任务，需要低延迟投递、明确路由、消费重试与死信，RabbitMQ 成本较低。Kafka 更适合高吞吐事件日志、长时间保留和多消费者回放。若需求变成行为流和事件分析，Kafka 更合适；不能只用“RabbitMQ 更快”这种空话。

### Q37：为什么不用 RocketMQ？

**参考答案：**

> RocketMQ 在顺序消息、延迟消息和大规模交易消息场景有优势，但当前没有复杂延时等级或超大规模吞吐，RabbitMQ 已满足路由、重试和 DLQ。选型主要考虑现有学习栈、部署成本和问题规模，不宣称 RabbitMQ 全面优于 RocketMQ。

### Q38：为什么不用 Redis Stream？你已经有 Redis 了。

**参考答案：**

> Redis Stream 可以让 Lua 原子完成秒杀预扣和入流，能解决 Redis 到 MQ 的双写窗口，适合关键预订命令。代价是把缓存、资格和队列故障域集中在 Redis，并要自己治理 pending、claim 和保留。规划任务已经以 MySQL 为事实源，RabbitMQ 更像唤醒通道；两个场景可以做不同选择。

### Q39：为什么不直接轮询数据库，MQ 是不是多余？

**参考答案：**

> 规划链路实际上已经采用数据库 Dispatcher，MySQL Attempt 是可靠事实源，MQ负责及时唤醒 Worker。纯轮询可以工作但延迟和数据库扫描压力更高；纯 MQ 又缺少业务级可恢复状态。两者结合是 durable state + transport 的取舍。

### Q40：Dispatcher 先把状态改成 DISPATCHED 再发 MQ，发失败怎么办？

**参考答案：**

> `claimForDispatch` 先用 CAS 将 Attempt 从 PENDING 改为 DISPATCHED；同步 `convertAndSend` 抛异常时会释放回 PENDING 并延迟重试。若进程在 claim 后、Broker 确认前崩溃，定时任务会扫描仍为 QUEUED + DISPATCHED 的陈旧记录并重置；若消息实际送达又被重置，消费端 `claimForExecution` 的状态 CAS 只允许一个消费者执行。当前没有把 confirm/return 回调写回 Dispatcher 状态。

### Q41：你配置了 publisher confirm，就能保证不丢吗？

**参考答案：**

> 不能。confirm 只表示 Broker 是否接受，return 表示 mandatory 消息是否无法路由，不代表消费者处理或 DB 提交。当前规划链有 MySQL Dispatcher 可恢复；秒杀链路没有持久待投递记录，而且业务尚未消费 confirm/return 回调，所以仍存在双写窗口。

### Q42：consumer ACK 和 publisher confirm 有什么区别？

**参考答案：**

> confirm/return 是生产侧到 Broker；consumer ACK 是消费方法成功后的消费侧确认。ACK 前事务已提交但连接断开会重复投递，因此消费者必须幂等。

### Q43：你的重试会不会把业务错误反复打三次？

**参考答案：**

> 应区分可恢复和永久错误。规划执行只对 DataAccess、超时、连接与 IO 等瞬时异常进入受限重试；候选非法、规则拒绝等直接失败。秒杀监听当前把多类 RuntimeException 都重试，分类仍需细化。

### Q44：DLQ 是垃圾桶吗？消息进去就完事了？

**参考答案：**

> 不是。DLQ 需要告警、查看原因、幂等重放和最终处置。当前秒杀死信消费者若补偿失败后吞异常，会 ACK 丢失，生产化应进入 parking queue 或失败表；规划死信也需要运维消费和状态对账。

### Q45：多实例 Dispatcher 会重复发送吗？

**参考答案：**

> 多实例都会扫描，但 `claimForDispatch` 使用带旧状态条件的 UPDATE，只有受影响行数为 1 的实例能发布。重复消息仍可能因 ACK/网络产生，所以消费端 `claimForExecution` 再做一次状态 CAS。

### Q46：为什么 MQ 消息只传 runId 和 attemptId，不把完整上下文塞进去？

**参考答案：**

> MySQL 是事实源，只传标识能避免消息中上下文过期，也降低敏感数据泄露和消息大小。Worker 收到后重新查询最新业务事实。代价是多一次数据库读取，但状态一致性和可恢复性更清晰。

### Q47：消息积压怎么办？

**参考答案：**

> 先监控 ready/unacked、最老消息年龄、消费耗时和失败率，再判断是模型慢、数据库慢还是消费者不足。规划任务还要限制全局模型并发和供应商配额，不能盲目扩 Worker；秒杀可以水平扩消费者，但数据库写入能力是上限。

## 7. 第六轮：数据库设计

### Q48：为什么有 Room、Run、Attempt、Event 四张表？是不是过度设计？

**参考答案：**

> Room 是长期协作聚合；Run 是用户发起的一次规划流程；Attempt 是一次可重试的实际执行；Event 是可观察、可补发的轨迹。一次 Run 可以因澄清产生多个 Attempt，SSE 断线又不能只依赖当前状态，所以职责不能简单合并。

### Q49：Run 和 Attempt 为什么不放一张表加 retry_count？

**参考答案：**

> retry_count 只能描述技术重试，不能保留每次执行的模型名、prompt 版本、错误和时间。澄清后的新 Attempt 使用了新的业务事实，也不是同一次网络重试。当前不保存完整 Prompt 正文，这避免重复存敏感偏好，但若要做精细回放，还需保存脱敏输入快照或哈希。分表能保留审计历史，同时 Run 提供稳定的用户视角。

### Q50：为什么 Event 还要落 MySQL？SSE 直接推不行吗？

**参考答案：**

> SSE 会断线，应用也会重启。事件先落 MySQL，SSE 只在事务提交后推送，前端可用 sequence/Last-Event-ID 补发。否则断线期间的事件永久丢失，也无法审计。

### Q51：`PlanEvent.sequence` 怎么保证多实例唯一？

**参考答案：**

> 当前实现用 JVM `synchronized` 后查询该 Run 最大 sequence 加一，只能保护单实例；多实例仍可能生成相同值，数据库唯一键会拒绝其中一个，但代码没有重试。这是已知缺口。可以使用 Run 行 `SELECT ... FOR UPDATE` 维护 next_sequence、数据库原子 UPDATE returning 语义，或插入冲突后重试。

### Q52：为什么状态用 VARCHAR，不用 tinyint？

**参考答案：**

> VARCHAR 可读、排障和迁移直观，不会因枚举序号调整破坏历史数据，代价是空间和比较成本更高。当前规模下可接受；应通过 Java enum、长度限制和状态迁移 SQL控制合法值，必要时增加 CHECK。

### Q53：表里有 version，但你真的用乐观锁了吗？

**参考答案：**

> 不能笼统说用了。当前关键规划迁移主要通过 `WHERE old_status = ...` 的状态 CAS，多表 JOIN UPDATE 保证只有合法旧状态能转换；部分 version 字段只是为后续乐观锁预留。面试时必须区分“存在 version 列”和“所有更新都校验 version”。

### Q54：为什么 Preference 同时存 raw_text、draft_json、confirmed_json？

**参考答案：**

> 原文用于审计和重新解析；draft 是模型或 fallback 生成、允许用户编辑的建议；confirmed 才是成员确认的规划事实。三者分开避免模型解析结果未经同意就成为硬约束输入。

### Q55：为什么 JSON，不全拆成列？

**参考答案：**

> 偏好和解释字段结构会演进，当前主要按 room/user 整体读写，JSON 能降低早期迁移成本。经常用于过滤、排序和索引的字段，例如 roomId、status、shopId 必须独立列；若未来需要按预算或过敏原做大量 SQL 查询，应抽取规范化列或生成列索引。

### Q56：为什么没有外键？

**参考答案：**

> 当前迁移使用应用约束和唯一索引，没有声明 FK，便于种子数据和演进，但会增加孤儿记录风险。生产化要根据删除策略决定：核心 Room/Run/Attempt 可加 FK 或使用明确软删除与审计任务，不能把“微服务都不用外键”当答案。

### Q57：数据库有哪些关键唯一约束？

**参考答案：**

> 邀请码唯一；房间成员 `(room_id,user_id)` 唯一；偏好 `(room_id,user_id)` 唯一；Attempt `(run_id,attempt_no)` 唯一；Event `(run_id,sequence)` 唯一；Proposal `(run_id,proposal_rank)` 唯一。旧秒杀订单和关注表反而缺少业务唯一索引，是当前需要补的缺口。

### Q58：Dispatcher 查询为什么需要联合索引？

**参考答案：**

> 当前查询同时按 status、dispatch_status、next_dispatch_at 过滤并按 id 限量扫描，现有 `(dispatch_status,next_dispatch_at)` 能先缩小待投递范围，但 status 是残余条件，`next_dispatch_at IS NULL OR <= now` 和 `ORDER BY id` 也可能阻止完整利用索引。数据量上来后应比较 `(dispatch_status,status,next_dispatch_at,id)` 等候选，并用真实数据分布和 EXPLAIN ANALYZE 决定；运行超时扫描则使用 `(status,update_time)`。

### Q59：为什么模型调用不能放在数据库事务里？

**参考答案：**

> 模型可能几十秒，长事务会占连接、持有锁、扩大回滚和死锁风险。执行链使用短事务 CAS 认领，事务外调用模型，再用短事务 CAS 提交 Proposal、Run、Attempt、Room 和 Event；晚到结果因旧状态不匹配而被丢弃。

### Q60：确认方案时两个人同时点怎么办？

**参考答案：**

> 只有房主有权限；即便重复点击，更新 Room 时要求旧状态是 `PLANS_READY` 且 confirmedProposalId 为空，只有一次更新受影响行数为 1。后续请求得到“已确认或状态变化”，这是状态 CAS 的幂等控制。

### Q61：房间满员判断先 count 再 insert，会不会超员？

**参考答案：**

> 会有并发窗口。`uk_room_user` 只能防同一用户重复加入，不能限制总人数。生产化应锁 Room 行或通过原子计数条件更新 `member_count < max_members`，然后插入成员；失败回滚整个事务。

## 8. 第七轮：秒杀一致性

### Q62：Lua 到底保证了什么？

**参考答案：**

> 只保证 Redis 内库存检查、重复用户判断和预扣在一次脚本中原子执行。它不保证 Redis、RabbitMQ 和 MySQL 原子提交，也不能替代数据库唯一索引。当前脚本成功路径甚至没有显式 `return 0`，Java 把 null 默认解释成成功，这是需要先修复的协议缺陷。

**继续追杀：** “你的 `seckill.lua` 成功路径没有 `return 0`，为什么现在看起来还能成功？”

> Redis 会返回 nil，而当前 Java 把 null 留在默认值 0，于是被误判为成功。这是偶然可用，不是正确协议；null 还可能代表脚本或反序列化异常，不能当成功。脚本应显式返回状态码，Java 用枚举完整处理成功、库存不足、重复购买和活动未初始化，未知值一律失败并告警。

### Q63：Redis 扣成功，MQ 没发出去怎么办？

**参考答案：**

> 当前捕获同步发送异常后执行 Lua 补偿，但进程在两步之间宕机或发送结果不确定仍有窗口。可让 Lua 同时写 Redis Stream/待投递记录，由 Dispatcher 重放；补偿必须按 orderId CAS，不能只按 userId。

### Q64：为什么补偿只按 userId 有问题？

**参考答案：**

> 旧订单补偿晚到时，用户可能已经重新预占一笔新订单。只删除 userId 会错误释放新预订。资格应保存 `userId -> orderId,state`，补偿比较当前 orderId 一致才删除并加库存。

### Q65：为什么有 Redisson 锁还会重复订单？

**参考答案：**

> 锁可能失效、Redis 可能丢数据、旁路代码可能绕开锁。数据库必须增加 `UNIQUE(voucher_id,user_id)`；锁只是降低同一用户并发冲突，不是最终不变量。

### Q66：为什么锁 key 只用 userId？

**参考答案：**

> 当前会让同一用户购买不同券也串行，粒度偏粗。更准确是 userId + voucherId。`tryLock()` 零等待失败也不应直接等同重复购买，应进入短暂重试或让数据库唯一约束决定。

### Q67：活动开始结束在哪里校验？

**参考答案：**

> 当前 Lua 没有校验 begin/end/status，旧 Java 校验被注释，这是缺口。生产化把活动元数据一起预热到 Redis，Lua 使用服务端时间判断，并在 DB 消费端二次校验。

**继续追杀：** “MySQL 条件扣库返回 false，你为什么正常 return？这条消息会怎样？”

> 当前代码会正常结束消费并自动 ACK，而且没有释放 Redis 预占，最终出现“无订单但资格和库存被占用”。数据库异常应抛出并进入受限重试；条件扣库为 false 属于 Redis 与 MySQL 库存漂移，应记录失败状态，按 orderId 做 CAS 释放并进入对账，不能静默返回。

### Q68：接口返回 orderId 是不是代表下单成功？

**参考答案：**

> 不是。它只表示 Lua 资格预占被当前 Java 代码解释为成功，且 `convertAndSend` 没有同步抛异常；不代表 Broker 已 confirm，更不代表 MySQL 已创建订单。更准确应返回 `202 Accepted + reservationId`，并提供状态查询区分 RESERVED、CONFIRMED、RELEASED。

### Q69：Redis 重启或数据丢失怎么办？

**参考答案：**

> 恢复时应根据 MySQL 活动库存和已确认订单重建 Redis，并在恢复期间关闭活动入口。当前数据库还没有 `(voucher_id,user_id)` 唯一约束，不能靠它防重复单；生产化必须先补该约束，再结合 Redis 持久化和定时对账，而且不能把 Redis 持久化当成绝对不丢。

### Q70：Redis Cluster 下脚本能直接跑吗？

**参考答案：**

> 多 key 必须同 slot。应传完整 KEYS，并使用 `{voucherId}` hash tag，例如 `stock:{123}`、`order:{123}`。当前动态拼 key 的脚本不适合直接迁移。

### Q71：怎么证明零超卖？

**参考答案：**

> 固定 1000 库存，执行大量不同用户和同用户重复请求；在 Redis 预扣后、发布后、DB 提交后 ACK 前分别杀进程；静默期后核对初始库存、Redis 余量、MySQL 扣减和唯一订单数。只报接口 QPS 不能证明一致性。

## 9. 第八轮：缓存、Redis 与 Redisson

### Q72：你说治理了穿透、击穿、雪崩，分别用什么？

**参考答案：**

> 空值缓存处理重复不存在 ID；热点逻辑过期、异步重建和互斥处理击穿；雪崩还需要 TTL 抖动、容量和故障降级，当前固定 TTL 尚未完整解决。不能把三个名词一次性打包成“全部解决”。

### Q73：逻辑过期为什么比普通 TTL 好？

**参考答案：**

> 热点过期后仍返回旧值，让单个后台任务刷新，用短期陈旧换稳定延迟。必须补 hard TTL 和最大陈旧边界，否则旧数据可能永久存在。

### Q74：你当前热点缓存第一次成功，第二次为什么可能 500？

**参考答案：**

> 热点 miss 按 RedisData 读取，降级路径却向同一 key 写裸 Shop；下一次再按 RedisData 解析会缺 expireTime。修复原则是一 key 一 schema，统一 CacheEnvelope，热点和普通策略只能改变期限，不能改变格式。

### Q75：更新数据库后删除缓存就一致了吗？

**参考答案：**

> 当前删除发生在事务提交前，并发读可能在删除后、提交前读旧 DB 并回填。即使 after-commit 删除，旧异步重建也可能晚到覆盖。需要 after-commit 失效加 generation fence，或者接受明确的短暂不一致并度量。

### Q76：你的缓存锁安全吗？

**参考答案：**

> 当前写固定值 1、10 秒过期并无条件 DEL；旧任务超时后可能删除新持有者的锁。应使用唯一 token + Lua 比较删除，或使用 Redisson，并让获取和释放发生在同一线程。

### Q77：为什么不用 Redisson 统一所有锁？

**参考答案：**

> 可以统一，能减少自研锁错误。但仍要考虑锁粒度、等待时间、watchdog、业务线程切换和 Redis 故障。缓存重建更适合 single-flight + 有界异步刷新，锁不是全部答案。

### Q78：线程池满了怎么办？

**参考答案：**

> 当前 AbortPolicy 会让 submit 抛异常，而且锁释放在异步任务 finally 内，任务没提交时只能等 TTL。应由 Spring 管理 Executor，监控队列和拒绝；拒绝时安全释放锁并返回旧值或走受控同步刷新。

### Q79：Caffeine 为什么还不能写到简历？

**参考答案：**

> 因为配置存在不等于业务能力。必须实际接入 L1 读写、容量与统计、多实例失效和短 TTL 兜底，再通过命中率和 DB 回源次数验证。

### Q80：Redis 挂了直接查 DB 不就行了？

**参考答案：**

> 无限 fail-open 会让缓存故障变数据库故障。普通查询可以短超时后走有 bulkhead 的 DB 降级；热点可保留短时 L1 last-known-good；秒杀入口应 fail-closed，不能绕过资格闸门。

### Q81：Redisson 和 StringRedisTemplate 为什么同时存在？

**参考答案：**

> StringRedisTemplate 用于数据结构和 Lua，Redisson用于高层分布式锁。当前 Redisson 地址硬编码 localhost，可能和 Spring Redis 指向不同实例，生产化必须统一配置源、认证、database 和超时。

## 10. 第九轮：登录、安全与 Feed

### Q82：验证码为什么能重复登录？

**参考答案：**

> 当前校验后不删除验证码，2 分钟内能反复换 token。应使用 GETDEL 或 Lua 原子消费，并限制手机号、IP、设备频率和失败次数。

### Q83：36000 分钟会话 TTL 是认真设计的吗？

**参考答案：**

> 这是学习版的过长滑动 TTL，不应作为安全设计。生产化同时设置 idle 和 absolute expiry，仅在剩余 TTL 低于阈值时续期，并支持设备会话和全端撤销。

### Q84：为什么不用 JWT？

**参考答案：**

> 当前选择 opaque token + Redis，是因为现有实现需要滑动续期和按 token 主动退出；它也便于未来增加即时撤销和设备会话。JWT 可减少服务端会话查询，但撤销和权限变更仍需额外状态机制。当前尚未实现设备管理、全端撤销和绝对过期，不能把这些说成现有能力。

### Q85：ThreadLocal 有什么风险？

**参考答案：**

> Servlet 线程复用，未清理会串用户。当前 afterCompletion 清理，但 preHandle 写入身份后 Redis 续期异常时可能未覆盖；更稳妥用最外层 Filter try/finally，并在请求开始先 remove。

### Q86：博客为什么能偷登录 token？

**参考答案：**

> 后端原样保存正文，前端用 v-html，token 又在 sessionStorage，可形成存储型 XSS 会话窃取。默认纯文本，富文本需服务端 allowlist、前端 sanitizer 和 CSP；会话可迁移 HttpOnly Cookie 并补 CSRF。

### Q87：Feed 为什么用 ZSet？

**参考答案：**

> member 是 blogId、score 是发布时间，支持倒序范围和时间游标。相同毫秒需要 maxTime + offset，否则包含边界会重复，排除边界会漏读。

### Q88：大 V 发一条内容会怎样？

**参考答案：**

> 当前同步遍历粉丝写扩散，发布延迟和部分失败会放大。生产化通过事务 Outbox 异步 fanout，普通用户推模式，大 V 拉模式或推拉结合，并监控投递 lag。

### Q89：点赞为什么会计数漂移？

**参考答案：**

> 先查 Redis、再改 MySQL count、再改 ZSet 不是原子操作，并发两次可能 DB +2 而 ZSet 只有一个 member。应建立 blog_like 事实表和唯一索引，接口改成幂等 PUT/DELETE，Redis 作为读模型。

## 11. 第十轮：Spring、Java 与 MyBatis 基础深挖

### Q90：为什么 `@Transactional` 自调用会失效？

**参考答案：**

> Spring 默认通过代理拦截外部方法调用，同对象 `this.method()` 不经过代理。当前秒杀通过懒注入 self 代理调用事务方法，更清晰的做法是拆独立 Transactional Service。

### Q91：RuntimeException 和 checked Exception 回滚有什么区别？

**参考答案：**

> Spring 默认对 RuntimeException/Error 回滚，checked Exception 默认不回滚；需要时使用 `rollbackFor`。更重要的是不要捕获异常后正常返回，否则事务可能提交；若转换业务异常，要保留回滚语义。

### Q92：Redisson watchdog 是什么？

**参考答案：**

> 未指定 leaseTime 时，持锁客户端会定期续期，避免业务超过默认租约提前释放。客户端宕机后续期停止，锁最终过期。它不能解决错误锁粒度、死循环续期和 Redis 故障。

### Q93：RedisIdWorker 如何组成 ID？

**参考答案：**

> 高位是相对起始时间的秒偏移，低 32 位是 Redis 按业务和日期 INCR 的序列。它支持多实例，但依赖 Redis和时钟，需考虑回拨、每天 2^32 上限和计数 key 清理。

### Q94：MyBatis-Plus 有什么缺点？

**参考答案：**

> 单表 CRUD 快，但直接暴露 IService 容易绕过业务边界，字符串字段、N+1 和复杂 SQL 也容易失控。关键命令应封装明确服务方法，复杂查询使用 Mapper SQL、索引和 EXPLAIN。

### Q95：为什么批量加载成员而不是循环查用户？

**参考答案：**

> 循环查询产生 N+1，网络往返和连接占用随成员数增长。先收集 userId 批量查询，再 Map 组装，数据库访问次数固定。博客列表目前仍有类似 N+1，是可改进点。

### Q96：`synchronized` 能解决分布式并发吗？

**参考答案：**

> 只能保护同一 JVM。PlanEvent sequence 当前就是本地保护，多实例需数据库锁、原子序列或冲突重试；秒杀消费端实际使用 Redisson，但仍缺数据库唯一约束，不能把分布式锁当作最终不变量。

## 12. 第十一轮：系统设计与扩展

### Q97：为什么不拆微服务？

**参考答案：**

> 当前规模和单人开发不值得承担服务发现、跨服务事务、追踪和部署成本。模块化单体便于事务和快速迭代；当团队边界、独立扩容或故障隔离需求明确时，再按 identity、content、reservation、planning 拆分。

### Q98：多实例部署后 SSE 为什么可能不实时？

**参考答案：**

> emitter 存在单机内存。如果浏览器连实例 A，Worker 在 B 落事件，B 只会推给自己的连接；A 能从 MySQL 补历史，但不会立即收到跨节点通知。需要 Redis Pub/Sub、RabbitMQ fanout 或网关 sticky session；无论如何 MySQL Event 仍是补发事实源。

### Q99：多实例后还有哪些隐患？

**参考答案：**

> 当前多实例隐患包括 PlanEvent 本地 sequence 锁、Redisson 地址硬编码、静态缓存重建线程池、本地 SSE emitter，以及定时扫描的重复执行。Dispatcher 扫描已有数据库 CAS；Caffeine 目前未接入业务链路，只有后续启用 L1 时才需要补多实例失效机制。

### Q100：系统容量上限在哪里？

**参考答案：**

> 秒杀入口可能先受 Redis 单线程 Lua 和数据库订单写限制；Agent 主要受模型供应商并发、延迟和成本限制；Feed 写扩散受粉丝数量；缓存故障受 DB bulkhead。容量分析应分链路，不用一个“QPS”概括整个系统。

### Q101：怎么做可观测性？

**参考答案：**

> 当前主要有 orderId/runId 关联日志和持久化 Agent Event，还没有完整 Micrometer 指标、Trace 和告警看板。生产化时，秒杀看预占、重复拒绝、消息 lag、确认/释放和对账差额；缓存看 fresh/stale/negative hit、DB amplification、重建队列和 stale age；Agent 看 Run 成功率、工具次数、规则拒绝、模型延迟、Token 与成本。日志不能记录 token、验证码和敏感偏好原文。

### Q102：怎么测试，不要只说 JUnit？

**参考答案：**

> 当前已有 Policy、DTO、工具预算、草稿指纹、Dispatcher CAS 和重复消费的单元/Mock 测试，以及少量 Spring Boot/MockMvc 测试；仓库还没有 Testcontainers、压测报告和故障注入。补强时用 Testcontainers 验证 Redis Lua、MySQL 唯一约束和 RabbitMQ 重投，API 测权限与状态契约，压测验证并发不变量，故障注入覆盖 Redis 预扣后、MQ 发布后、DB 提交后 ACK 前，以及 Agent 超时和晚到结果。

### Q103：如果只允许你再做三件事，做什么？

**参考答案：**

> 第一修安全红线和缓存双格式；第二给订单/关注补唯一约束并为秒杀增加可重放待投递与对账；第三建立压测、故障注入和 Agent 评测集。先让简历主张有证据，再扩新功能。

### Q104：项目最有价值的设计到底是什么？

**参考答案：**

> 项目最有价值的方向，是把不可信或异步组件放进可验证边界：规划链中 MySQL Run/Attempt 与 Java 状态机保存事实，MQ 负责唤醒；模型只提供语义建议，最终由 Java Policy 校验。秒杀链当前仍有 Redis-MQ 双写、按 userId 补偿和数据库唯一约束缺失等窗口，价值在于能明确这些缺口并给出可验证的补齐方案，而不是宣称所有链路已经可恢复。

## 13. 压轴：面试官故意否定你

### Q105：你这项目漏洞不少，凭什么算亮点？

**参考答案：**

> 学习项目的价值不是假装已经生产化，而是能发现 Happy Path 之外的问题并给出可验证改造。我能准确指出哪些已经实现、哪些是缺口，以及如何用数据库约束、状态 CAS、可重放投递和故障实验补齐，这比只会复述教程更能体现工程能力。

### Q106：你说了很多“生产化会怎么做”，是不是其实都没做？

**参考答案：**

> 我会明确分层回答：当前实现、已验证能力、下一步方案。简历只保留当前代码能支撑的主语，未完成项不会说成结果；但面试官问到边界时，我必须能给出生产化设计。下一步会按 P0/P1 清单逐项落代码和测试，而不是只停留在文档。

### Q107：如果我现在让你删掉一半技术栈，你删什么？

**参考答案：**

### Q108：最后一句，为什么我要招你？

**参考答案：**

> 我不只会把框架接起来，还会追问它在并发、重复、超时和故障下是否成立。我能把业务事实、缓存、消息和模型的责任边界说清楚，并愿意用约束、测试和指标把设计落到可验证结果。

## 14. 十个答错即挂的句子

下面这些话不要说：

1. “Lua 保证了整个秒杀系统强一致。”
2. “用了 Redisson 就绝对不会重复下单。”
3. “RabbitMQ 开了 confirm，所以消息绝对不丢。”
4. “Redis 挂了直接查数据库就行。”
5. “用了 Caffeine 依赖，所以是多级缓存。”
6. “结构化 JSON 说明模型不会幻觉。”
7. “Spring AI 比 LangGraph 更先进。”
8. “SSE 可以保证业务事件不丢。”
9. “MySQL 可重复读可以替代唯一索引。”
10. “这个项目支持群组投票和否决重规划。”

## 15. 十个救场句式

当你被追到不会时，用这些结构，不要瞎编：

1. “我先限定一下这个保证的范围：它只发生在 Redis/单事务/单实例内。”
2. “当前版本解决了正常链路，但在 X 和 Y 之间仍有双写窗口。”
3. “这个替代方案可行，我选择当前方案主要基于业务规模和集成成本，不是绝对优劣。”
4. “数据库约束是最终防线，应用检查只负责提前失败和降低冲突。”
5. “这项能力目前是设计，不是已完成结果，我不会把它写成上线事实。”
6. “这个数字需要压测口径支撑，我可以说明测试方案，但不应该现场估算结果。”
7. “模型输出只是建议，Java 校验通过后才成为业务事实。”
8. “MQ 是传输通道，业务状态仍由 MySQL 保存。”
9. “这个多实例问题单机测试看不出来，需要用数据库 CAS/共享事件或故障实验验证。”
10. “我目前没有可靠答案，我会先查看实际调用链和官方文档，再决定设计。”

## 16. 练习方式

1. 每次只练一轮，先遮住答案口述。
2. 每题回答控制在 30-60 秒，面试官追问后再展开。
3. 每个“保证”后主动补一句“保证范围是……”；每个“最终一致”后补“由谁重试、状态在哪、多久收敛”。
4. 把不能回答的题标记到代码对应类，实际打开代码走一遍。
5. 完成代码修复或压测后，把答案中的“生产化会”改成“我已经，并通过……验证”。
