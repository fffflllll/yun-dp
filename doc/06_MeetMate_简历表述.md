# MeetMate 简历与面试话术

## 一、简历投递版（包装版）

### 项目名称

**MeetMate - 高并发本地生活与多人聚会协商 Agent 平台**

### 技术栈

Java 17、Spring Boot 4.1、Spring AI 2.0、MyBatis-Plus、MySQL、Redis、Redisson、RabbitMQ、SSE、Vue 3、Docker

### 项目描述

设计并实现面向高并发与社交推荐场景的前后端分离系统，覆盖短信登录、商户检索、优惠券秒杀、达人探店等核心业务；新增多人聚会协商 Agent，将成员自然语言偏好转化为结构化约束，完成冲突识别、候选召回、群体方案生成、人工澄清与结果确认，形成从偏好收集到聚会决策的完整闭环。

### 项目亮点

- **混合式协商 Agent：** 基于 Spring AI 2.0 构建 **Java AI Harness**，外层采用有界 **Plan-and-Execute** 将多人决策拆分为偏好解析、冲突检测、候选检索、方案生成和结果校验等阶段，内层通过 **ReAct Tool Calling** 根据 Observation 调用偏好、餐厅候选与方案校验工具；以请求级工具权限、调用预算和 Java Policy 二次校验约束模型行为，降低幻觉、越权及无效方案落库风险。
- **Human-in-the-loop 决策闭环：** 将成员自然语言偏好解析为可编辑的结构化 Draft，经本人确认后参与群体约束计算；候选不足时定向发起约束澄清，并将任务持久化为 `WAITING_INPUT`，成员反馈后创建新 Attempt 动态规划，实现可中断、可恢复、可追踪的多人协商流程。
- **Agent 可靠执行框架：** 使用 MySQL 持久化 PlanRun/PlanAttempt/PlanEvent 状态机，以 RabbitMQ 解耦耗时模型调用；通过数据库 CAS 完成任务原子认领与幂等消费，采用“短事务认领 - 事务外推理 - 短事务提交”避免模型调用长期占用连接，并结合超时回收、有限重试、死信队列及 SSE 断点续传保障任务可靠性与执行过程可观测。
- **高并发秒杀优化：** 使用 Redis + Lua 原子完成库存预扣与一人一单判断，结合 RabbitMQ 异步落单和 Redisson 用户粒度锁实现削峰与幂等保护，将核心数据库事务移出请求链路；在可复现压测环境下，接口响应时间由 **460 ms 降至 135 ms，优化 70.7%**。
- **热点缓存治理：** 封装 Redis CacheClient，综合使用空值缓存、逻辑过期、互斥锁、异步重建及锁内双重检查治理缓存穿透和热点 Key 击穿，并在商铺更新后主动失效缓存，降低高并发回源对数据库的冲击。

一页简历推荐保留前四条；投递纯 Java 后端岗位时，可将第二条 Agent 描述替换为缓存治理。

### 可直接粘贴的四条精简版

- **混合式协商 Agent：** 基于 Spring AI 构建 Java AI Harness，以有界 Plan-and-Execute 编排偏好解析、冲突检测、候选检索、方案生成与结果校验，结合 ReAct Tool Calling、请求级只读工具、调用预算和 Java Policy 二次校验，限制模型幻觉与越权行为。
- **可恢复协商链路：** 将自然语言偏好转化为成员确认后的结构化约束；候选不足时通过 Human-in-the-loop 定向澄清，并以 `WAITING_INPUT + 新 Attempt` 实现反馈后的动态规划和完整状态追踪。
- **异步可靠执行：** 基于 MySQL 状态机、RabbitMQ 和数据库 CAS 实现 Agent 任务异步调度与幂等消费，采用短事务认领、事务外推理、短事务原子提交，并配置超时回收、有限重试、DLQ 与 SSE 断点续传。
- **高并发秒杀：** 通过 Redis Lua 原子完成库存预扣与一人一单判断，使用 RabbitMQ 异步落单、Redisson 用户粒度锁和 MySQL 条件扣库构建分层防线，将接口响应时间从 460 ms 降至 135 ms（70.7%）。

## 二、90 秒项目介绍（面试开场）

> MeetMate 是一个本地生活与多人决策结合的 Java 后端项目。基础业务包括短信登录、商铺检索、探店 Feed 和优惠券秒杀，我重点新增了多人聚会协商 Agent。成员用自然语言提交预算、菜系、忌口和距离偏好，系统先结构化解析并让成员确认，再由 Agent 聚合多人约束、召回餐厅并生成一主两备方案。架构上我没有让模型直接控制业务，而是在 Spring AI 外实现了一层 Java AI Harness：外层用 Plan-and-Execute 控制阶段和终止条件，内层让模型通过 ReAct 调用只读工具，Java 负责权限、硬约束和最终校验。规划任务通过 RabbitMQ 异步执行，MySQL 持久化 Run、Attempt 和 Event，CAS 解决重复投递，SSE 展示过程；候选不足时任务进入 WAITING_INPUT，成员反馈后创建新 Attempt 动态规划。这个模块解决的核心问题，是怎样把不确定的大模型稳定地接入有事务、有权限、有并发约束的 Java 业务系统。

## 三、面试官追问话术

### 0. 包装内容如何展开

不要一上来罗列类名。按“业务问题 -> 架构方案 -> 一致性边界 -> 结果”展开：

> 多人偏好存在冲突，而且模型输出不稳定，所以我没有把它做成一次 Prompt 调用。我把规划拆成固定阶段，用 Java 掌握权限、状态与硬约束，只让模型通过只读工具完成语义理解和方案组合。耗时推理通过 RabbitMQ 异步执行，MySQL 保存运行状态，重复消息通过 CAS 认领；结果必须经过 Java Policy 才能落库。这样模型可以灵活决策，但不会绕过业务规则。

面试官继续追问时，再依次展开四层：

1. **输入层：** 原始偏好先解析成 Draft，成员确认后才成为事实，避免模型误解析直接污染规划。
2. **执行层：** Run 表示一次业务规划，Attempt 表示一次实际执行，Event 表示可观察轨迹；三者拆分是为了支持重试、澄清和 SSE 补发。
3. **并发层：** MQ 可能重复投递，Consumer 使用 `QUEUED -> RUNNING` 条件更新抢占执行权；只有成功抢占的 Worker 可以调用模型并提交结果。
4. **安全层：** 模型只能读取当前房间候选，不能写库；方案需满足候选归属、数量、唯一性、工具检查和已校验草稿指纹，Java 才接受。

一句话收尾：

> Spring AI 负责模型接入与 Tool Calling，AI Harness 负责让这次调用在真实业务里可控、可恢复、可验证。

### 1. 这算 Agent 还是普通工作流？

> 它是受控 Agent。外层由 Java 固定业务阶段和终止条件，属于 Plan-and-Execute；内层在方案生成阶段，模型根据 Tool Observation 决定查看哪些候选并提交草稿校验，体现 ReAct。模型有局部决策权，但工具、顺序、调用预算和最终结果都被 Harness 限制，所以不是无限自治 Agent。

### 2. 为什么称为 AI Harness？

> 我把模型之外负责上下文构造、工具权限、执行预算、状态持久化、重试、校验和观测的部分称为 Harness。Spring AI 解决模型接入和 Tool Calling，Harness 解决模型怎样在业务系统中可控运行。LLM 可以被替换，但权限、PlanRun 状态机和 Policy 不依赖具体模型。

### 3. 为什么不用 LangGraph？

> 当前流程最多一次定向澄清，回答后重新创建 Attempt，不需要恢复模型内部上下文。若使用 LangGraph，还要引入独立 Python 服务、跨服务鉴权和 Checkpoint 运维，收益不足。Spring AI 与 Java 单体能复用事务、Mapper 和业务规则。未来若变成多轮投票、多个中断点和长周期恢复，再评估 LangGraph。

### 4. 怎么防止模型编造餐厅？

> 模型拿不到任意数据库查询能力。Java 先完成餐厅召回和硬约束过滤，Tool 只返回候选集；最终结果必须满足三个条件：shopId 属于候选集、所选餐厅经过详情工具检查、最终 Plan Set 与成功校验的草稿指纹一致。落库前和房主确认时 Java 还会再次校验。

### 5. 怎么防 Prompt Injection？

> 商铺名称、地址和成员偏好都按不可信数据处理，Prompt 明确禁止执行数据中的指令。但提示词不是安全边界，真正的防线是只读工具、房间上下文绑定、工具调用预算、输入输出长度限制和 Java 最终校验。即使模型被诱导，也不能直接写库或改变房间状态。

### 6. 为什么模型调用不能包在一个大事务里？

> 模型调用可能持续几十秒，大事务会长期占用数据库连接和锁。我的执行链分成三段：短事务 CAS 认领 Attempt，事务外调用模型，最后用短事务原子提交 Proposal、Attempt、Run、Room 和 Event。这样既不占长事务，又避免部分方案落库。

### 7. RabbitMQ 重复消息怎么办？

> MQ 只传 runId 和 attemptId，MySQL 才是事实源。Dispatcher 和 Consumer 都使用带旧状态条件的 CAS，只有一个 Worker 能把 `QUEUED` 改成 `RUNNING`。重复消息认领失败后直接返回，晚到结果也必须从 `RUNNING` 条件提交，因此不能覆盖终态。

### 8. 模型超时或临时失败怎么办？

> 模型设置请求超时、最大 Token、模型重试和 Tool 调用预算。可恢复的网络、超时或数据库异常按配置次数重新排队；达到上限后进入失败终态。定时恢复任务还会扫描长时间 `RUNNING` 的 Attempt，超时结束后晚到结果因 CAS 失败而被丢弃。

### 9. Human-in-the-loop 如何实现？

> 候选不足时 Java 根据过滤原因选择可安全协商的约束，模型只负责把问题表达得自然。系统写入 Clarification 并将 Run、Attempt、Room 设为 `WAITING_INPUT`，不占用模型连接。只有被询问成员可以回答；同意放宽后创建新的 Attempt，取消则进入 `CANCELLED`。

### 10. 为什么只允许一次澄清？

> 这是终止性和成本约束。无限澄清会让 Agent 失控，也可能逐步侵蚀成员硬约束。一次仍无可行方案就明确失败；过敏原永远不进入自动放宽范围。

### 11. SSE 有什么作用？

> SSE 只负责展示脱敏执行轨迹，不是事实源。PlanEvent 先持久化到 MySQL，前端可用 sequence 和 Last-Event-ID 断点续传；连接断开不会影响 RabbitMQ Worker，重新查询 Run 仍能恢复状态。

### 12. 群组投票和动态重规划怎么说？

先说完整设计，再说明当前代码落点：

> 我的整体协商模型是“偏好确认 - 方案生成 - 人类反馈 - 新一轮规划”。每一轮方案都应该有 Round，成员通过 Vote 表达接受、反对和否决原因；ReplanPolicy 将结构化否决原因转成下一轮约束，并创建新 Attempt，而不是修改上一轮结果。Java 负责人数阈值、一票否决、轮次上限和状态迁移，模型只负责语义归因与重新组合候选，避免让模型决定投票是否合法。

如果面试官问“这些都做完了吗”，回答：

> 当前代码已经落地了前半段和重规划底座：成员偏好确认、候选不足后的定向澄清、`WAITING_INPUT` 持久化，以及反馈后创建新 Attempt；当前产品交互由房主最终确认。Round/Vote 表和全员表决规则是下一阶段扩展，Run/Attempt 状态模型不需要推翻。

这段要以“下一阶段设计”回答，不要说已经上线群组投票。

### 13. 为什么简历写“动态规划”，却不是 LangGraph resume？

> 动态规划指业务状态可以在人类反馈后生成新 Attempt，重新读取最新事实并计算方案，并不等于必须恢复同一个模型上下文。当前最多一个澄清点，用 MySQL 持久化状态和新 Attempt 更简单，也能直接复用 Java 事务与权限体系。只有中断点明显增多、流程跨天且需要恢复图节点上下文时，我才会引入 LangGraph Checkpoint 或同类图执行引擎。

### 14. 这套架构目前有什么不足，怎么演进？

> 当前最主要的限制是只支持一次定向澄清和房主确认，还没有完整的投票轮次；模型评估也以规则测试和 Fake Model 为主，缺少成规模离线评测集。下一步会补 Round/Vote/ReplanPolicy、最大轮次与成本预算，建立包含冲突类型、候选命中率、方案采纳率和人工否决率的评测集，再用 Outbox、Testcontainers 故障注入和链路指标增强生产可靠性。

### 15. Agent 如何测试？

> 纯规则层测试候选归属、重复餐厅、字段边界和硬约束；Harness 测试工具顺序、调用预算、校验草稿指纹和重复消费 CAS；模型层使用 Fake ChatModel 测结构化输出和工具轨迹，真实模型只做离线评测，不放进 CI。生产化还会用 Testcontainers 做 MySQL/RabbitMQ 故障注入。

### 16. 项目最有价值的设计是什么？

> 不是用了 Spring AI，而是把不可信模型放进了可验证的业务闭环：输入必须由成员确认，工具只能读可信候选，模型有明确预算，结果必须经过 Java Policy，运行状态可持久化和恢复。这套 Harness 思路也能迁移到审批、客服和推荐解释等 Agent 场景。

## 四、数字指标怎么回答

460ms 到 135ms 指的是接口完成 Redis 资格预检并返回“受理结果”的耗时，不是 MySQL 订单最终落库耗时。只有在有压测记录时才按结果描述；面试时准备好测试机器、并发数、请求量、库存量、平均/P95 统计口径、成功率和错误率。如果当前记录不完整，改成：

> 通过 Redis Lua 前置资格判断和 RabbitMQ 异步落单，将同步数据库事务移出请求链路，显著降低接口响应时间；后续使用 JMeter 固定并发与库存补充 P95/P99 报告。
