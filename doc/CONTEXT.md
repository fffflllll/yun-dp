## 决策树（grilling 记录）

### D1 — 根节点：MeetMate 的北极星（2026-07-12）
- **结论**：A — 学习 / 作品级 Demo。
- **含义**：第一优先级不是"真能上线的餐厅产品"，而是**证明"业务系统守门、AI 只做语义"这套分层能落地、可测、可替换**。
- **v1 成功标准（初拟）**：端到端跑通 + 关键并发 / 澄清链路有自动化测试；而非真实用户订位量。
- 候选：B=真能用小产品 / C=单纯练手 LangGraph / D=其他。已排除 B、C。

### D2 — 承重概念："可替换"指什么（2026-07-12，已被 D3 覆盖/降级）
- 初答 A1：整个 Python Agent 服务可替换，Java 契约是稳定边界。
- **已被 D3 推翻**：用户明确"可替换没用，就是想给简历加亮点"。故"可替换"不再是项目目标，无需为它做证明（如第二个 stub 实现）。

### D3 — 真正的根节点：简历亮点（2026-07-12，推翻 D1/D2）
- **结论**：用户原话"我就想给简历加点亮点"。"学习/作品级 Demo"只是外壳，**真实动机 = 一个能写进简历、显得厉害的 AI Agent 作品**。
- **后果**：
  - 砍掉"做第二个实现去证明可替换"（B1 否决）。
  - 但**契约 / 状态机 / 幂等 / 评测 / 可观测性这套重机器保留**——它不是为了证明可替换，而是简历上的"工程成熟度"硬证据。
  - 衡量标准从"架构哲学是否被证明"变成"招聘方看了觉不觉得厉害、能不能快速看懂价值"。
- **开放子问题（真正要 grill 的）**：
  - 简历面向什么角色？（后端 / AI 工程师 / 全栈）——决定强调点。
  - 必须真能跑 / 有 live demo，还是"架构漂亮 + 部分实现"就够？
  - 最该让人记住的"wow"是哪个：LangGraph 多节点编排？群体冲突 + HITL 澄清？系统-of-record 与 AI 分层？
  - MVP 最薄但仍有亮点的纵切在哪？

### D4 — 简历面向角色（2026-07-12）
- **结论**：E2 — AI / Agent 工程师向。
- **含义**：简历主场 = LangGraph 多节点编排 + HITL 澄清挂起/恢复 + 语义冲突解释 + 评测体系。Java 系统-of-record 作为"我懂怎么把不可信 AI 安全嵌进严肃业务"的佐证。
- 排除：E1(后端主场) / E3(全栈) / E4(海投，已归入 E2 打法)。

### D5 — 是否真跑通 / 演示形态（2026-07-12）
- **结论**：F2 — 必须接**真实 LLM** 端到端跑，演示即真实效果。
- **风险已告知**：抽风 / 烧钱 / 超时 / 录屏翻车风险高；用户已知并选择接受。
- 含义：MVP 硬指标 = 真实 LLM 下端到端跑通一次；自动化测试仍可用 Fake LLM（见 Python 文档 §26.2），但"演示产物"用真模型。

### D6 — 最该突出的"wow"元素（2026-07-12）
- **结论**：G2 — AI 知道"什么时候该问人"：群体偏好冲突 + 信息缺失时，Agent 主动挂起、请 Java 落 clarification、等人回答后 `resume` 接着跑。带人类在环的状态机。
- **定位**：LangGraph 多节点状态机（G1）是其底层机制，藏后面当技术支撑，不单独当门面；分层哲学（G3）作佐证。
- 简历落点：摘要第一句 / Demo 第一帧 = "一个会主动问人的多人在环 Agent"。

### D7 — MVP 最薄纵切（2026-07-12）
- **结论**：H1 — 1 房间 + 2 冲突成员(A 火锅/D 不吃) + 1 次真 LLM 规划 + 1 次澄清挂起/恢复 + 出方案。
- **砍掉（推 v2）**：投票、重规划、多轮、完整评测集；仅留 Fake-LLM 单测保底。
- **前端**：不建设完整产品前端，但建立**最小 Vue3 Demo Shell**（`web/`，单页演示）；不做通用组件库 / 完整样式系统 / 正式页面迁移。精力压在 Agent 主链。（覆盖早前"不新造 Vue 工程"措辞）
- **规模预期**：几百行核心即可 demo 出 G2 全貌。

### D8 — 项目来源（2026-07-12）
- **结论**：I1 — 继续基于 hmdp / yun-dp 增量改造，在现有 `dp` 仓库 Java 上挂 Agent 链路，复用现成房间/用户/商铺表。
- **叙事**："在真实业务系统上安全接入 AI 层"，Java 系统-of-record 成加分项。hmdp 是公认教学底座无妨，加分项在原创的 AI 增量层。
- 排除：I2(greenfield，丢"嵌入现有系统"故事) / I3(独立干净模块，前期重构成本)。

### D9 — 技术栈锁定程度（2026-07-12，已被 D10 推翻）
- **结论**：J1 — LangGraph 锁死，其余弹性。
- **理由**：G2 的挂起/恢复人类在环状态机正是 LangGraph 招牌能力（`interrupt` + `Command(resume=...)` + checkpointer），免费拿到最难部分；且是简历高频关键词。LLM 供应商 / Java 内部实现 / 前端保持弹性，能跑即可。
- 排除：J2(全锁，堆砌且脆弱) / J3(全弹，写不出框架名，E2 减分)。

### D10 — Agent 技术路线改为 Spring AI 2.0（2026-07-16）
- **结论**：MVP 改用 **Spring AI 2.0**，规划 Agent 与既有 Spring Boot 应用部署在同一个 Java 进程中。
- **推翻**：D9 的“LangGraph 锁死”不再有效；原 Python Agent Service、跨服务 Tool API、LangGraph checkpoint 与 `interrupt / resume` 不再属于当前 MVP。
- **理由**：当前业务主链是“成员确认偏好后发起一次受控规划”，Spring AI 的模型调用、结构化输出与 Tool Calling 足以承载，单体方案更容易真正完成和演示。
- **保留**：Java 继续拥有用户、房间、偏好、候选商铺、状态与最终业务事实；LLM 只做语义归纳、方案组织与解释，不直接写业务数据。
- **升级约束**：Spring AI 2.0 引入前，必须独立完成 Spring Boot 4 兼容升级和回归验证，不能把框架升级与 Agent 业务开发混成一次不可诊断的改动。

### D11 — 聚会方案输出形态（2026-07-16）
- **结论**：一次规划生成一个聚会方案集，其中包含 **1 个首选方案 + 2 个备选方案**。
- **方案内容**：每项至少包含建议时间、目标店铺、集合安排、推荐理由、满足的成员偏好，以及为了群体可行性让步的软偏好。
- **交互含义**：首选方案用于直接确认；房主可改选任一备选。LLM 不替成员确认最终方案。
- **约束**：三项必须绑定 Java 提供且校验通过的候选数据，不能编造店铺、营业时间、距离或成员偏好。

---

### D12 — 成员偏好的录入与确认（2026-07-16）
- **结论**：成员先用自然语言描述偏好，Spring AI 将其提取为结构化偏好，成员检查、修改并确认后才提交。
- **事实边界**：自然语言原文和模型提取结果都是偏好草稿；只有成员本人确认的结构化结果才是规划输入。
- **偏好内容**：至少覆盖预算、时间、餐饮类型、忌口/过敏、距离或交通方式，并区分不可违反的硬约束与可让步的软偏好。
- **失败处理**：模型调用失败或结果无法校验时保留原文并允许成员手动填写，不能阻塞房间基本流程。

---

### D13 — 冻结成员集合后再开始规划（2026-07-16）
- **结论**：采用“房主锁定成员名单 → 锁定成员全部确认偏好 → 房主手动开始规划”的顺序。
- **锁定含义**：锁定前邀请码有效；锁定后停止加入，参与本次规划的成员集合固定，系统才有明确的“全员”。
- **启动条件**：只有锁定集合中的每位成员都存在已确认偏好时，房主才能开始规划；模型调用不得由最后一位成员确认偏好的动作自动触发。
- **规则替换**：MVP 不使用“达到最低提交人数即可规划”的 `minSubmittedMembers` 规则，不能静默跳过未确认成员。
- **异常处理**：成员未到齐时由房主继续等待或在锁定前移除；锁定后的退出、重新开放名单等操作不进入当前 MVP。

---

### D14 — 无可行候选时进行一次定向澄清（2026-07-16）
- **结论**：MVP 支持至多一次定向澄清。Java 硬约束过滤得到零个候选时，由规划 Agent 根据结构化冲突生成一个面向具体成员的问题。
- **回答选项**：被询问成员可以将指定硬约束降为软偏好，或取消本次规划；房主不能代替其他成员放宽其硬约束。
- **运行语义**：创建澄清后结束当前模型调用并持久化规划任务为 `WAITING_INPUT`，不保持 HTTP 连接或 JVM 线程等待。成员回答后发起一次新的规划调用。
- **终止条件**：一次澄清后仍无可行候选，则以可解释的“无可行方案”结束，不继续递归询问或擅自忽略硬约束。
- **责任边界**：候选为空和约束是否被放宽由 Java 判定；LLM 只负责解释冲突并把问题表述得自然、明确。

---

### D15 — LLM 供应商通过 OpenAI 兼容配置注入（2026-07-16）
- **结论**：规划 Agent 使用 Spring AI 的 OpenAI 兼容 ChatModel，不把供应商、域名或密钥写死在代码中。
- **配置项**：通过环境变量注入 `MEETMATE_AI_API_KEY`、`MEETMATE_AI_BASE_URL`、`MEETMATE_AI_MODEL`，分别映射到 Spring AI 的 API Key、Base URL 与模型名配置。
- **默认策略**：Base URL 可以指向 OpenAI、DeepSeek 或其他兼容服务；真实密钥只存在本地/部署环境，示例配置使用空值或占位符。
- **测试策略**：自动化测试替换为 Fake ChatModel，不访问外部模型、不消耗配额。

### D16 — MVP 聚焦餐厅聚会（2026-07-16）
- **结论**：MVP 的“聚会规划”专指多人聚餐，Agent 只从现有 `tb_shop` 及其 MeetMate 元数据中生成餐厅方案。
- **候选事实**：店铺名称、类型、地址、坐标、均价、评分、营业时间与治理后的餐饮标签由 Java 提供并校验。
- **明确不做**：MVP 不抽象电影、桌游、户外活动等通用活动类型，也不让 LLM 编造未在候选集中的地点或活动。
- **后续扩展**：其他活动作为新的候选类型和规则集单独设计，不改变餐厅规划的业务契约。

---

### D17 — MVP 由房主确认方案（2026-07-16）
- **结论**：成员可以查看方案集，但 MVP 由房主在首选和两个备选中选择一项确认。
- **Java 守门**：确认时 Java 重新校验店铺、营业时间、候选归属与硬约束，校验通过后才写入已确认方案。
- **明确不做**：MVP 不实现成员投票、否决轮次、投票截止、平票处理或投票驱动的重规划；这些属于后续版本。
- **术语边界**：Agent 生成的是建议方案集，房主选择并经 Java 校验后的条目才是已确认方案。

---

### D18 — 异步规划通过 SSE 推送进度（2026-07-16）
- **结论**：规划仍然创建持久化 `PlanRun` 并在后台执行，前端通过 SSE 订阅该 Run 的实时事件；SSE 是观察通道，不是业务状态来源。
- **事件边界**：至少推送 `RUN_STARTED`、候选召回/过滤进度、`WAITING_INPUT`、`RUN_COMPLETED` 和 `RUN_FAILED`；断线后以前端重新查询 Run 状态为准。
- **结果边界**：模型生成的结构化 JSON 必须完整解析并经 Java 校验后，才通过 `RUN_COMPLETED` 发送方案集并落库；任意中间文本或 token 都不能直接成为业务结果。
- **可靠性**：RabbitMQ/后台 worker 负责执行和重试，SSE 断开不影响 Run；同一 Run 只能产生一个最终结果。

---

### D19 — 展示可解释执行轨迹，不展示原始思维链（2026-07-16）
- **结论**：用户可以看到 Agent 的工作阶段和可验证的决策摘要，但不接收模型 token、Prompt、隐藏推理文本或未经脱敏的 Tool 参数。
- **阶段事件**：SSE 推送固定阶段，例如读取已确认偏好、检查硬约束、召回餐厅、计算候选排序、组织方案、Java 最终校验。
- **摘要事件**：每阶段可附带简短事实说明，例如“召回 20 家，因预算过滤 6 家，因忌口过滤 2 家，剩余 12 家”；摘要只能引用 Java 已确认数据。
- **结果边界**：执行轨迹用于解释和调试展示，不参与业务判定；最终方案仍以 Java 校验后的结构化结果为准。

---

### D20 — 规划流程、执行尝试与事件日志分层（2026-07-16）
- **结论**：以 `PlanRun` 表示一次房间规划流程，以 `PlanAttempt` 表示每次实际模型执行，以 `PlanEvent` 表示可断线补发的安全执行轨迹。
- **PlanRun**：持有房间级状态 `QUEUED / RUNNING / WAITING_INPUT / SUCCEEDED / FAILED`、当前尝试号和已确认方案引用；同一房间同时只能有一个活跃 Run。
- **PlanAttempt**：记录 attempt 编号、模型/Prompt 版本、执行状态、耗时和错误；成员澄清或可重试失败都会产生新的 Attempt，但不会新建用户可见的 Run。
- **PlanEvent**：使用单调 `sequence` 记录阶段、摘要、澄清、完成和失败事件；SSE 以 `runId + sequence` 支持 `Last-Event-ID` 断线补发。
- **消息边界**：RabbitMQ 只负责投递待执行 Attempt；MySQL 是 Run、Attempt、Event 的事实来源，SSE 断开不影响任务执行。

### D21 — Spring Boot 升级目标与依赖边界（2026-07-16）
- **结论**：为接入 Spring AI 2.0，应用目标升级到 Spring Boot 4.1.0，Java 17 保持不变。
- **同步升级**：MyBatis-Plus 切换到 `mybatis-plus-spring-boot4-starter:3.5.17`，Redisson 升级到 `4.6.1`；移除手动钉死的旧 `mybatis-spring` 依赖，让 Starter 管理兼容版本。
- **构建仓库**：项目级 Maven 仓库使用官方 Maven Central，避免未来版本被过时镜像阻断。
- **验证边界**：框架升级必须先通过现有功能编译与可控测试，再验收 MeetMate Agent；旧 `VoucherOrderControllerTest` 的千线程登录测试不作为升级通过条件，需单独修复。

### D22 — 房间消息纳入聊天室 MVP（2026-07-16）
- **结论**：MeetMate 房间包含成员可见的持久化消息区；成员只能读取和发送自己所在房间的消息。
- **事实来源**：消息正文、发送者和时间由 Java 写入 MySQL，不能只保存在浏览器会话中。
- **交互取舍**：MVP 使用 REST 增量拉取，前端按消息 ID 轮询；不在本轮引入 WebSocket、未读数、撤回或消息审核。
- **边界**：聊天室消息用于成员协作和补充偏好，不会直接改变已确认偏好或规划状态。

### D23 — 规划 Agent 采用受控 ReAct + Java Plan-and-Execute（2026-07-16）
- **结论**：规划阶段采用 Java 编排的有限执行计划，并在方案组织阶段使用 Spring AI 请求级 Tool Calling；不是单次无工具 LLM 调用，也不是允许模型任意循环和写业务状态的全自治 ReAct。
- **必要工具**：读取当前房间已确认偏好、按 Java 硬约束查找候选餐厅、查看候选详情、校验方案草稿。真实 AI 模式缺少任一必要调用即判定 Attempt 失败。
- **权限边界**：工具实例绑定当前 Room / Run / Attempt，且全部只读；确认方案、放宽约束、修改偏好、改变房间状态等写操作不暴露给模型。
- **双重校验**：模型可调用校验工具修正草稿，最终结构化结果仍由 Java 策略服务再次校验后才能落库。
- **可观测性**：计划和工具执行以脱敏 PlanEvent 展示；不保存或展示原始思维链、Prompt、Token 与未脱敏工具参数。

---

### D24 — AI Harness 作为当前 Agent 边界（2026-07-16）
- **结论**：当前项目把包围 Spring AI 调用的 Java 确定性执行边界称为 **AI Harness**；它负责可信上下文、工具权限、状态迁移、重试、最终校验、持久化和安全轨迹，LLM 只是其中的非确定性组件。
- **执行约束**：Planning Tool Calling 由请求级工具实例承载，Java 代码限制工具顺序、调用次数、模型超时和输出规模，并要求最终返回的 Plan Set 必须是已成功校验的同一份草稿。
- **重试语义**：瞬时执行失败在同一 Attempt 内按上限重新排队；模型执行超时结束旧 Attempt 并恢复房间可重新开始；成员定向澄清仍创建新的 Attempt，不恢复旧模型上下文。
- **简历边界**：当前 MVP 不宣称 LangGraph、成员投票、否决轮次或投票驱动的动态重规划；这些能力只有在状态机、接口和集成测试实际落地后才可写入项目经历。
